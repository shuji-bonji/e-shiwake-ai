# MCP クライアント実装計画

## 1. 背景と課題

### 現状のアーキテクチャ

```
┌─────────────────────────────────┐    ┌─────────────────────────────────┐
│       e-shiwake (PWA)           │    │      e-shiwake-ai               │
│  SvelteKit + IndexedDB          │    │  MCP Server + SQLite            │
│                                 │    │                                 │
│  ┌───────────────────────────┐  │    │  ┌───────────────────────────┐  │
│  │ WebMCP (12ツール)          │  │    │  │ MCP Server (23ツール)      │  │
│  │ navigator.modelContext    │  │    │  │ stdio トランスポート        │  │
│  │ Chrome 146+ 限定           │  │    │  │ Claude Desktop等から利用   │  │
│  └────────┬──────────────────┘  │    │  └────────┬──────────────────┘  │
│           │                     │    │           │                     │
│  ┌────────▼──────────────────┐  │    │  ┌────────▼──────────────────┐  │
│  │ Dexie (IndexedDB)         │  │    │  │ better-sqlite3 (SQLite)   │  │
│  └───────────────────────────┘  │    │  └───────────────────────────┘  │
│                                 │    │                                 │
│  ❌ 接続なし                      │    │  ❌ 接続なし                     │
└─────────────────────────────────┘    └─────────────────────────────────┘
```

**問題点**: 2つのシステムが完全に分離しており、データベースも別々。

### ユーザーの声（UIの必要性）

> AIの入力の手軽さ（音声・テキスト）は魅力だが、出力がテキストだと：
>
> - 一覧との対比がないので不安
> - 日付順のどこに挿入されたかイメージしにくい
> - 試算表や損益計算書など、帳簿の種類を知らない人はプロンプトで想像できない
> - 決まったフォーマットの帳簿はUIで見た方がわかりやすい

**結論**: AI入力 + UI出力 のハイブリッドが最適解。

### 方針決定

| 選択肢                            | 説明                        | 判定                                |
| --------------------------------- | --------------------------- | ----------------------------------- |
| A. PWA内にMCPクライアント組み込み | 既存PWAにAIチャットUIを追加 | ⭐ **採用**                         |
| B. 別アプリとして構築             | 新規SvelteKitアプリ         | ❌ 重複が多い                       |
| C. MCP Server経由でSQLite統一     | PWAのDBをSQLiteに移行       | ❌ IndexedDB→SQLiteはブラウザで困難 |

**理由**: 既存PWAの帳簿UIが既に完成しており、そこにAI入力機能を追加するのが最も効率的。データソースもIndexedDBに統一し、WebMCPの仕組みを活用する。

## 3. 実装アプローチ

### Phase 0: 基盤整備（WebMCPの進化）

現在の WebMCP は Chrome 146+ の `navigator.modelContext` に依存しているが、これはブラウザ提供のAIモデルにツールを公開する仕組み（サーバー側）。

MCPクライアントとして動作するには、**別のアプローチ**が必要。

#### 3つのAI統合パターン

```
パターン1: WebMCP（現状）
  ブラウザ内蔵AI → navigator.modelContext → e-shiwakeツール → IndexedDB
  ※ Chrome 146+ 限定、モデル選択不可

パターン2: Anthropic API 直接呼び出し
  ユーザー入力 → Anthropic API (tool_use) → e-shiwakeツール → IndexedDB
  ※ APIキー必要、最も柔軟

パターン3: ローカルMCPサーバー経由
  ユーザー入力 → MCPクライアント → MCP Server (stdio) → SQLite
  ※ Node.js必要、PWAの制約と合わない
```

**採用: パターン2（Anthropic API 直接呼び出し）を主軸 + パターン1を併用**

### 理由

- **パターン2**: APIキーは必要だが、Claude の tool_use で高品質な仕訳解釈が可能。Webアプリから直接呼べる
- **パターン1**: Chrome 146+ ユーザーはAPIキー不要で利用可能（無料枠）
- **パターン3**: PWA（ブラウザ）から stdio プロセスは起動できないため不採用

## 6. 技術的な検討事項

### APIキーの安全性

| リスク           | 対策                                 |
| ---------------- | ------------------------------------ |
| APIキーの漏洩    | localStorage暗号化（Web Crypto API） |
| ネットワーク盗聴 | HTTPS必須（GitHub Pages は対応済み） |
| XSS攻撃          | CSP設定、入力サニタイズ              |
| 課金超過         | 月額上限設定、使用量表示             |

**重要**: フロントエンドからの直接API呼び出しはAPIキーがブラウザに露出するリスクがある。
以下の緩和策を検討：

1. **CORS対応**: Anthropic API は `anthropic-dangerous-direct-browser-access` ヘッダーで直接呼び出しを許可している
2. **ユーザー責任**: 個人のAPIキーを個人のブラウザで使用する形態
3. **暗号化保存**: Web Crypto API で localStorage のAPIキーを暗号化

### トークン使用量の最適化

```
システムプロンプト: ~2,000トークン（勘定科目一覧含む）
ツール定義:         ~1,500トークン（12ツール）
会話履歴:          ~500トークン/往復
───────────────────────────────────
1回の仕訳入力:     ~4,000トークン（約$0.01〜$0.03）
```

**月100件の仕訳**: 約$1〜$3（Sonnet 4.5 利用時）

### オフライン対応

PWA のオフライン要件との兼ね合い：

```
オンライン時: AI入力 + UI表示（フル機能）
オフライン時: UI手入力のみ（既存機能で完全動作）
              AIパネルは「オフラインです」表示
```

## 8. e-shiwake-ai との関係

### 役割分担の明確化

```
e-shiwake (PWA)
├── UI（帳簿・仕訳帳・レポート）       ← メイン
├── WebMCP（ブラウザ内AIツール）        ← Chrome 146+
├── AI チャット（Anthropic API直接）   ← 新規追加
└── IndexedDB（データストア）           ← 単一データソース

e-shiwake-ai (MCP Server)
├── Claude Desktop / Cowork 統合       ← 外部AIクライアント用
├── SQLite（独立データストア）          ← CLIベース
└── E2Eテスト基盤                     ← テスト用
```

### データ同期は行わない

PWA（IndexedDB）と MCP Server（SQLite）のデータ同期は**意図的に行わない**。

**理由**:

- 2つは異なるユースケース（ブラウザ vs CLI）
- 同期は複雑性を大幅に増加させる
- JSON エクスポート/インポートで十分な移行が可能
- PWA のオフライン・ファースト原則を維持

## 9. 補足: MCP Server の今後

e-shiwake-ai の MCP Server は以下の用途で引き続き有用：

1. **Claude Desktop / Cowork**: MCP設定で直接利用（現状通り）
2. **E2Eテスト**: MCPツール経由のテスト基盤
3. **バッチ処理**: CLI からの一括データ操作
4. **将来の HTTP トランスポート**: MCP Server を HTTP 対応にすれば、PWA から直接呼べるようになる可能性

### 将来的な統合の可能性

```
将来構想（HTTP MCP トランスポート対応時）:

  e-shiwake PWA
       │
       ▼  HTTP/SSE
  e-shiwake-ai MCP Server（HTTP トランスポート）
       │
       ▼
  SQLite（単一データソース）
```

MCP SDK が HTTP Streamable トランスポートを安定サポートした段階で、
IndexedDB → SQLite への移行を検討する価値がある。

```
│  │  🤖 表示しました         │  │  → 試算表に切替       │   │    │
│  │     ↓                  │  │  → 損益計算書        │   │    │
│  │                        │  │  → 貸借対照表        │   │    │
│  └────────────────────────┴────────────────────────────┘    │
│                                                             │
│  ← アーティファクトが「使い捨て」ではなく「固定アプリ」に →          │
└─────────────────────────────────────────────────────────────┘
```

この構想では、Claude Desktop のアーティファクト右ペインが「固定的なアプリUI」として機能する。
チャットで指示すると右側の帳簿ビューがリアルタイムに更新される世界。

**現時点での制約**:

- アーティファクトは毎回使い捨て（固定化できない）
- UI操作（クリック、フォーム入力、D&D）はアーティファクト内で閉じる
- アーティファクトから MCP ツールを呼び出す手段がない
- Claude Desktop のプラグイン機構が未成熟

### 業界の動向: AIの「居場所」競争

各社が「AIをどこに住まわせるか」で差別化を図っている：

```
Google       → ブラウザ（Chrome + Gemini）
               既存: 検索、Gmail、Docs、YouTube...
               戦略: ブラウザ = AI の居場所。WebMCP で全Webアプリが
                     AIのツールになる

GitHub       → コードエディタ（VS Code + Copilot）
               既存: リポジトリ、CI/CD、Issues...
               戦略: 開発者の作業場 = AI の居場所

Grok         → SNS（X / 旧Twitter）
               既存: リアルタイム情報、ソーシャルグラフ
               戦略: 情報の流れる場所 = AI の居場所
               ※ トラフィック増加はSNS連携の恩恵

Vercel       → デプロイ基盤（v0 → Webアプリ生成）
               既存: Next.js、デプロイインフラ
               戦略: アプリの生まれる場所 = AI の居場所

Anthropic    → ???（Claude Desktop / Cowork）
               既存: なし（プラットフォームを持たない）
               戦略: MCP で「どこからでも繋がれるプロトコル」を標準化
                     プラットフォームがないなら、接続レイヤーで勝負
```

**Anthropic の課題と戦略**:

Anthropic だけ「既存プラットフォーム」を持っていない。
だからこそ Claude Desktop / Cowork 自体をプラットフォームにしようとしている。
MCP は「ブラウザもエディタも持っていないからこそ、接続の標準を握る」という戦略。

Cowork の登場は、この文脈で理解できる：
ファイル操作、スケジュール管理、ドキュメント作成...
日常のタスクを AI が直接扱える場所を作ろうとしている。

### 表示と操作の境界

AIクライアントで「どこまでできるか」の現実的な整理：

| 機能             | 表示のみ（読み取り） | プロンプト代替              | UI操作必須     |
| ---------------- | -------------------- | --------------------------- | -------------- |
| 試算表           | ✅ テーブル表示      | -                           | -              |
| 損益計算書       | ✅ テーブル表示      | -                           | -              |
| 仕訳一覧         | ✅ テーブル表示      | -                           | -              |
| グラフ・チャート | ✅ React/HTML        | -                           | -              |
| 仕訳の追加       | -                    | ✅ 自然言語で指示           | -              |
| 仕訳の検索       | -                    | ✅ 「Amazonの仕訳を探して」 | -              |
| 勘定科目の選択   | -                    | ✅ 「消耗品費で」           | ドロップダウン |
| 仕訳の編集       | -                    | 🟡 「3行目を5,000円に」     | インライン編集 |
| 証憑PDF紐付け    | -                    | -                           | ❌ D&D必須     |
| 日付範囲選択     | -                    | ✅ 「1月から3月」           | カレンダーUI   |

**結論**: 表示系はAI出力で十分。入力系はプロンプトで大部分カバー可能。
ファイル操作（D&D）だけは UI が必要。
